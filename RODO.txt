QUÉ ES ESTE PROYECTO
---------------------
AppPanacar es el sistema interno para un taller de chapería y pintura con dos
sucursales. Uso real: 6 computadoras y 6 celulares, no necesariamente
conectados al WiFi (hay que asumir que puede andar offline o con conexión
intermitente entre sucursales).

Este repo arrancó como copia de otro proyecto (AppEmar, un local de otro
rubro) que ya traía una arquitectura modular con Tkinter de interfaz. Esa UI
de escritorio no sirve para este proyecto: la idea es que el frontend final
sea HTML (para que ande en compus y celulares por igual sin instalar nada
nativo). Por eso se limpió todo lo que era Tkinter y quedó solo el backend
(SQLite + lógica de negocio), a la espera de que se diseñe el frontend HTML
y cómo se va a servir.

- Cada área (users, clients, products, branches) tiene su propio db.py /
  logic.py, todos compartiendo la misma DB SQLite.

  obtener_conexion() como context manager, con foreign_keys ON).
- Esquema de las 4 tablas, todas con columna status (1 = activo, 0 = borrado
  lógico) para no perder datos con un DELETE real.
- CRUD completo + validaciones de negocio en los 4 módulos, con las
  validaciones genéricas (email, teléfono, DNI, CUIT, fecha/edad mínima,

- Contraseñas de usuario hasheadas con pbkdf2_hmac + salt (nunca texto plano).

FASE DE IMPLEMENTACIÓN (dejar de vivir solo en localhost)
---------------------------------------------------------------------
Plan para probar rendimiento real antes de pagar un servidor: hostear
primero en la propia PC (Arch Linux, IP pública fija pedida al proveedor
de internet, no hace falta Dynamic DNS entonces), y después mudar lo mismo
a una instancia Amazon Linux en AWS sin rehacer nada. Notar que esto decide
a favor de "servidor central" la pregunta abierta más arriba de DB por
sucursal vs servidor central: mientras el servidor no sea alcanzable, las
sucursales quedan sin sistema.

Entorno aislado y portable (Docker):
  - Se recomienda Docker por dos motivos a la vez: aísla la app del usuario
    de escritorio de la PC personal (no correr Gunicorn con el usuario de
    todos los días) y permite mover el mismo contenedor a una instancia
    Amazon Linux sin rehacer cada paso a mano (Gunicorn, systemd,
    dependencias) en la máquina nueva. Falta armar: un Dockerfile (imagen
    base de Python, gunicorn como CMD, la app copiada adentro) y un volumen
    para data/database.db, para no perder la DB si se recrea el contenedor.

Servidor de aplicación:
  - Gunicorn en vez de flask run / python app.py: ese es el servidor de
    desarrollo de Flask, mono-hilo, pensado para no exponerse nunca a
    internet (tira ese warning en cada arranque).
  - systemd (o el proceso único dentro del contenedor) para que se reinicie
    solo si crashea y arranque con la PC.

Reverse proxy + HTTPS:
  - Caddy (pacman -S caddy en Arch): certificado Let's Encrypt automático,
    Caddyfile de pocas líneas apuntando a localhost:8000 (Gunicorn). Caddy
    redirige http -> https por default; confirmarlo igual.
  - Necesita un dominio (barato) con registro A apuntando a la IP fija.

Red:
  - Port forwarding en el router: 80 y 443 -> IP interna de la PC.
  - Firewall de la PC (nftables o ufw, ver archwiki): solo esos dos puertos
    abiertos hacia afuera, todo lo demás cerrado (incluido el puerto de SSH
    si se administra remoto — ver seguridad más abajo).

Seguridad a resolver EN EL PROYECTO (código) antes de exponerlo:
  - SECRET_KEY: hoy tiene fallback hardcodeado ('dev-panacar-secret-key' en
    src/config.py) si no está seteada la variable de entorno. Generar una
    real (python -c "import secrets; print(secrets.token_hex(32))") y
    setearla como variable de entorno del proceso/contenedor, nunca en el
    repo.
  - Cookies de sesión: create_app() no fuerza nada hoy. Sumar a la config de
    Flask: SESSION_COOKIE_SECURE=True (solo viaja por HTTPS),
    SESSION_COOKIE_HTTPONLY=True, SESSION_COOKIE_SAMESITE='Lax' (buscar
    "Flask session cookie security" en la doc oficial de Flask).
  - CREDENCIALES_PRUEBA.txt: ya está en .gitignore, pero antes de exponer la
    app hay que cambiar esas contraseñas de prueba (admin/panacar2026, etc.)
    por unas reales, quedaron pensadas solo para desarrollo local.
  - Debug mode: confirmar que nunca quede en True corriendo expuesto a
    internet — con la app pública, el debugger interactivo de Werkzeug
    permite ejecutar código arbitrario si algo tira un error 500, no es solo
    un detalle cosmético.

Seguridad a resolver EN LA PC / infraestructura (no es código del repo):
  - Usuario dedicado sin privilegios para correr Gunicorn (no el usuario de
    escritorio, no root) — o que quede resuelto solo por estar dentro del
    contenedor Docker.
  - SSH si se administra remoto: autenticación por clave (no por
    contraseña) y fail2ban contra fuerza bruta (archwiki: "OpenSSH",
    "fail2ban").
  - pacman -Syu regular: la PC va a estar expuesta a internet 24/7 mientras
    dure la prueba.
  - Backup de data/database.db: durante la prueba esta PC es el server
    real; un solo archivo SQLite sin backup es un punto único de falla. Un
    cron simple que lo copie a otro disco/nube todos los días alcanza para
    esta etapa.

¿Esto se hace una sola vez o cada vez que se modifica algo?
  - Gunicorn, systemd, Caddy, dominio, port forwarding y firewall son SETUP
    DE UNA SOLA VEZ: se arma y queda funcionando solo.
  - Lo que SÍ se repite cada vez que cambia el código es el "deploy": traer
    los cambios al servidor y reiniciar el proceso. Con Docker es reconstruir
    la imagen y reiniciar el contenedor; sin Docker sería git pull + pip
    install (si cambiaron dependencias) + reiniciar el servicio de systemd.
    Más adelante conviene un script chico (deploy.sh) que encadene esos
    pasos en vez de hacerlos a mano cada vez.

¿Postgres desde ya?
  - Todavía no. SQLite alcanza para esta etapa (2 sucursales, un puñado de
    dispositivos) y toda la conexión a DB ya está centralizada en
    src/db/connection.py, lo que acota el esfuerzo de migrar el día que
    haga falta. La señal real para migrar sería ver, durante esta misma
    prueba de rendimiento, errores de "database is locked" por escrituras
    concurrentes desde varias sucursales a la vez — ahí se migra con datos
    reales en la mano, no antes por las dudas.


VISIÓN A FUTURO (todavía sin diseñar, solo para no perder la idea)
---------------------------------------------------------------------
- Sucursales acotadas: cada usuario pertenece a una sola sucursal y responde
  en nombre de ella.
- Transferencias entre sucursales (efectivo o producto) sin poder modificar
  la otra sucursal directamente — surge del futuro módulo de ventas/
  movimientos.
- Mensajería entre sucursales tipo "archivos cifrados" para pedidos de
  actualización, conciliaciones o aprobaciones con descuento.
- Posible 2FA (TOTP con pyotp) para el login de Admin/BackOffice más
  adelante.
- Módulo de siniestros: ya tiene tile en la pantalla principal y una página
  placeholder (src/modules/administrar/siniestros/, blueprint 'siniestros',
  /siniestros) que solo avisa que está sin diseñar. Es la entidad real que
  une cliente + vehículo + aseguradora (no un dato suelto de cada uno por
  separado). Un mismo cliente puede tener varios siniestros, cada uno con su
  propia aseguradora y su propio vehículo. Necesita:
    - Un id legible para humanos (no solo el autoincrement de la tabla),
      pensado como un resumen de patente + DNI, para que se pueda
      identificar un siniestro de un vistazo sin abrir el registro.
    - Selector de cliente existente o alta de uno nuevo sin salir del
      formulario del siniestro.
    - Selector de vehículo existente o alta de uno nuevo (ver ítem
      siguiente) sin salir del formulario.
    - Selector de compañía de seguro (ya existe el catálogo
      insurance_companies, falta el módulo que lo consuma).
    - Selector de estado (ya existe el catálogo claim_statuses, falta el
      módulo que lo consuma).
    - El resto de los datos del siniestro (fecha, descripción, etc.)
      todavía sin definir.
- Tabla de vehículos: ya existe (src/modules/administrar/vehicles/), con
  marca -FK a vehicle_brands-, modelo, año, dominio, color, número de
  chasis y número de motor. Todavía sin dueño/cliente a propósito (ver
  arriba); falta decidir esa relación cuando se diseñe el módulo de
  siniestros, para que este último pueda referenciar "el Hilux de tal
  cliente" sin repetir esos datos sueltos cada vez.
